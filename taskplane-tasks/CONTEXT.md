## Technical Debt / Future Work

_Items discovered during task execution are logged here by agents._

- [x] **TP-033 direct socket operation conversion** — Core broker, worker, client, log-ingest, task-processor, socket-layer, frame-test, and demo socket open/bind/connect/send/receive/poll call sites now use `Zmqx.Monad` helpers under explicit context ownership. Remaining direct `Zmqx.*` references are intentional non-global operations: socket option/subscription setup, context runner creation (`withContext`/`runZmqx`), type names, and existing `Zmqx.EventLoop` commands deferred to TP-034.
- [x] **TP-032 explicit LotosApp ZMQ context migration** — `LotosEnv` now carries both `LoggerEnv` and an explicit `Zmqx.Context`; `LotosApp` implements `MonadZmqx`; `forkApp` and worker LogTransport EventLoop callbacks reuse the captured app context; TaskSchedule executables and ZMQ-facing log tests use `runZmqApp`. `runApp` remains as a compatibility wrapper for logger-only call sites, while raw direct-socket frame tests may continue to use `runZmqContextIO` until later migrations need app-level context ownership.
- [ ] **Evaluate remaining zmqx EventLoop/socket-ownership follow-ups** — TP-027 pinned `zmqx` to `v0.1.1.1` and established a green direct-API baseline; TP-028 removed the worker executor's fixed 10-second empty-queue sleep with a bounded STM wake-on-enqueue signal, added wake/count regression coverage, and removed the broker socket-layer `> poll` stdout spam; TP-029 moved the worker reliable-log DEALER behind a `Zmqx.EventLoop` transceiver with mailbox ACK handling and smoke coverage; TP-030/TP-031 deferred EventLoop for the worker backend loop and broker socket layer. TP-034 converted the worker reliable-log EventLoop to obtain its context through `MonadZmqx.askContext` and run with `Zmqx.EventLoop.withEventLoopIn`; TP-035 then moved the worker backend DEALER plus internal task-status PAIR behind a separate `Zmqx.EventLoop` that hands frames to unbounded STM queues for socket-loop-thread parsing, enqueue/wake, task-status forwarding, and heartbeat sends. EventLoop-registered sockets must be opened through the explicit monadic ZMQ context and run with that same context, then treated as worker-owned and not touched directly outside `EventLoop.sends`/`recv`. Remaining work should stay focused on optional EventLoop ownership for other runtime-loop sockets or hot paths without changing established multipart frame ordering unless covered by a dedicated follow-up test plan and benchmark proof.
- [x] **TP-031 decision:** keep direct polling in `LBS.SocketLayer` for broker frontend ROUTER, backend ROUTER, and TaskProcessor PAIR traffic. The direct SocketLayer thread already owns those sockets; EventLoop migration would add a second owner thread plus bounded mailbox/drop semantics around client ACKs, worker task dispatch, retry/garbage handling, scheduler notifications, and heavy queue/map mutations without measured fairness or latency gain. TP-031 documented this in source, removed a narrow ReaderT wrapper from task-status handling, kept ZMQ multipart shapes unchanged, and made the single-worker smoke use a run-local LogIngest journal by default so verification is repeatable.
- [x] **TP-030 decision:** keep direct polling in `LBW.socketLoop` for worker backend DEALER + internal PAIR transport. EventLoop migration was deferred due callback-in-loop pressure and shutdown/error-mode complexity (`stopped-loop`/`ETERM`) without measured latency gains; TP-035 later revisited this with lightweight EventLoop callbacks, socket-loop-thread queue drains, stopped-loop termination, and regression/smoke coverage.
- [x] **TP-035 worker backend EventLoop migration** — `LBW.socketLoop` now uses an EventLoop-owned backend DEALER with callback-to-STM handoff queues, bounded/fair queue draining, enqueue-before-wake task handling, periodic heartbeat sends through the backend EventLoop, and explicit stopped-loop error logging/termination. TP-036 replaced the internal task-status PAIR with an unbounded backend-owned STM handoff guarded by backend transport state, so task callbacks return/log `ETERM` promptly after backend shutdown rather than blocking on a stopped peer. Worker LogIngest remains on an independent DEALER/EventLoop; stopped log EventLoops terminate the log loop while ordinary send/ACK failures retry, and log queue drop markers remain visible without applying bounded/drop semantics to task/status traffic. `LotosApp` forked children are now cancelled and awaited before ZMQ context teardown. Verification covered worker frame/wake/log transport suites, `cabal build all --enable-tests`, and both single-/multi-worker smoke scripts. Remaining risk: backend task/status queues are intentionally unbounded to preserve no-drop semantics; add observability/bounds in a future production overload task if needed.
- [x] **TP-037 broker LogIngest EventLoop migration** — Broker `LogIngest` now registers its ROUTER as an explicit-context `Zmqx.EventLoop` transceiver and sends ACKs through the EventLoop while keeping decoding, journal append, retention, dedupe, and query-cache mutation on the LogIngest thread. The EventLoop callback only dispatches complete multipart frame sets into a bounded LogIngest-owned STM queue; queue-full dispatch increments observable rejected accounting and receives no ACK, preserving at-least-once worker retry semantics rather than claiming exactly-once delivery. Verification covered targeted log ingest/worker transport/protocol suites, `cabal build all --enable-tests`, and both single-/multi-worker smoke scripts.
