- [ ] **Evaluate remaining zmqx EventLoop/socket-ownership follow-ups** — TP-027 pinned `zmqx` to `v0.1.1.1` and established a green direct-API baseline; TP-028 removed the worker executor's fixed 10-second empty-queue sleep with a bounded STM wake-on-enqueue signal, added wake/count regression coverage, and removed the broker socket-layer `> poll` stdout spam; TP-029 moved the worker reliable-log DEALER behind a `Zmqx.EventLoop` transceiver with mailbox ACK handling and smoke coverage; TP-030/TP-031 initially deferred EventLoop for the worker backend loop and broker socket layer. TP-039 moved the broker SocketLayer frontend/backend ROUTERs and TaskProcessor PAIR traffic to explicit-context EventLoop ownership with unbounded callback-to-owner-thread queues for task/status traffic. EventLoop-registered sockets must be opened through the explicit monadic ZMQ context and run with that same context, then treated as worker-owned and not touched directly outside `EventLoop.sends`/`recv`. Remaining work should stay focused on optional EventLoop ownership for other runtime-loop sockets or hot paths without changing established multipart frame ordering unless covered by a dedicated follow-up test plan and benchmark proof.
- [x] **TP-031 decision:** keep direct polling in `LBS.SocketLayer` for broker frontend ROUTER, backend ROUTER, and TaskProcessor PAIR traffic. The direct SocketLayer thread already owns those sockets; EventLoop migration would add a second owner thread plus bounded mailbox/drop semantics around client ACKs, worker task dispatch, retry/garbage handling, scheduler notifications, and heavy queue/map mutations without measured fairness or latency gain. TP-031 documented this in source, removed a narrow ReaderT wrapper from task-status handling, kept ZMQ multipart shapes unchanged, and made the single-worker smoke use a run-local LogIngest journal by default so verification is repeatable.
- [x] **TP-030 decision:** keep direct polling in `LBW.socketLoop` for worker backend DEALER + internal PAIR transport. - [x] **TP-035 worker backend EventLoop migration** — `LBW.socketLoop` now uses an EventLoop-owned backend DEALER with callback-to-STM handoff queues, bounded/fair queue draining, enqueue-before-wake task handling, periodic heartbeat sends through the backend EventLoop, and explicit stopped-loop error logging/termination. TP-036 replaced the internal task-status PAIR with an unbounded backend-owned STM handoff guarded by backend transport state, so task callbacks return/log `ETERM` promptly after backend shutdown rather than blocking on a stopped peer. Worker LogIngest remains on an independent DEALER/EventLoop; stopped log EventLoops terminate the log loop while ordinary send/ACK failures retry, and log queue drop markers remain visible without applying bounded/drop semantics to task/status traffic. `LotosApp` forked children are now cancelled and awaited before ZMQ context teardown. Remaining risk: backend task/status queues are intentionally unbounded to preserve no-drop semantics; add observability/bounds in a future production overload task if needed.
- [x] **TP-037 broker LogIngest EventLoop migration** — Broker `LogIngest` now registers its ROUTER as an explicit-context `Zmqx.EventLoop` transceiver and sends ACKs through the EventLoop while keeping decoding, journal append, retention, dedupe, and query-cache mutation on the LogIngest thread. The EventLoop callback only dispatches complete multipart frame sets into a bounded LogIngest-owned STM queue; queue-full dispatch increments observable rejected accounting and receives no ACK, preserving at-least-once worker retry semantics rather than claiming exactly-once delivery. Verification covered targeted log ingest/worker transport/protocol suites, `cabal build all --enable-tests`, and both single-/multi-worker smoke scripts.
- [x] **TP-038 broker SocketLayer handler extraction** — Broker `LBS.SocketLayer` still owned frontend ROUTER, backend ROUTER, and TaskProcessor PAIR sockets on the direct `ZmqxM.poll` loop after TP-038, but decoded-message behavior was split into explicit helpers for client enqueue/ACK, load-balancer worker-task dispatch/bookkeeping, worker status heartbeat updates, and worker task-status retry/garbage/notify handling. Send paths remained explicit callbacks using the existing `toZmq` frame instances, so TP-038 was EventLoop preparation only and not a socket-ownership migration.
- [x] **TP-039 broker SocketLayer EventLoop migration** — Broker `LBS.SocketLayer` now registers the frontend ROUTER, backend ROUTER, TaskProcessor dispatch PAIR, and TaskProcessor notify PAIR in one explicit-context `Zmqx.EventLoop.withEventLoopIn` bracket. EventLoop callbacks only write complete multipart frames to unbounded STM queues; the SocketLayer app thread drains them with three-way frontend/backend/TaskProcessor rotation and invokes the TP-038 handlers so client ACK-after-enqueue, worker dispatch bookkeeping, heartbeat/liveness, retry/garbage, and scheduler notifications preserve their ordering. Client ACKs, worker tasks, and TaskProcessor notifications send through `EventLoop.sends`. Verification covered broker/client/worker frame and scheduler tests, `cabal build all --enable-tests`, and both TaskSchedule smoke scripts. Residual risk: the broker handoff queues are intentionally unbounded to avoid silent task/status loss; production overload observability or bounds remain future work if needed.
- [x] **TP-040 TaskProcessor EventLoop migration** — Broker `LBS.TaskProcessor` now registers its internal notification and worker-dispatch PAIR sockets in an explicit-context `Zmqx.EventLoop.withEventLoopIn` bracket. Notify reads use a bounded EventLoop mailbox as wake hints with the existing trigger timeout, and worker dispatches use sequential `EventLoop.sends`, so scheduler trigger placement, worker-task ordering, stale-worker recovery, and retry-ready requeue behavior remain unchanged. No direct `ZmqxM.receivesFor`/`ZmqxM.sends` runtime path remains in TaskProcessor; the PAIR sockets are opened only to connect and register them into the EventLoop. Verification covered worker frame/wake retry tests, `cabal build all --enable-tests`, and both single-/multi-worker TaskSchedule smoke scripts. Residual risk: the 1024-entry notify mailbox may drop newest wake hints under extreme notification bursts, but trigger timeout/counter scheduling still guarantees eventual processing.
- [x] **TP-043 overload observability** — Worker backend/status/task queues and broker SocketLayer/TaskProcessor handoff queues now maintain no-drop `HandoffQueueStats` snapshots with current depth, high-water depth, totals, and warning thresholds. Queue and metric updates are linearized in STM for production enqueue/drain paths so `currentDepth` does not drift under producer/consumer interleavings. Broker `/info` exposes `runtimeQueueStats`; worker custom status reporters receive `srHandoffQueueStats`; `/logs/stats` remains LogIngest-only rejected/drop accounting. Verification covered worker frame/wake, LogIngest separation, full build with tests enabled, and both TaskSchedule smoke helpers.
